Yes — and the answer is in how git hooks work, which is genuinely surprising.

**The core thing:** `.githooks/pre-commit` *being in the repo* doesn't make git run it. Git only runs hooks from `.git/hooks/` by default — and `.git/hooks/` is *per-clone*, not tracked. To make git use the repo's `.githooks/` directory, each clone has to opt in via:

```
git config core.hooksPath .githooks
```

That config is local to the clone (`git config` without `--global` writes to `.git/config`, also untracked). So:

- **First time it stopped working**: probably the original clone where this got set up. Then a fresh clone (new machine, deleted `.git/`, etc.) wiped it.
- **This time (s133's commit slipped through)**: same shape — this current clone never had `core.hooksPath` set.

You can verify with `git config --get core.hooksPath` (currently empty here). The reason git does it this way is *security* — hooks execute arbitrary shell, and git deliberately won't auto-run code committed to a repo without your explicit per-clone consent.

**Common fixes to make this less fragile:**

1. **Bootstrap script** (lowest friction): a `bin/setup` or `make install-hooks` that runs `git config core.hooksPath .githooks`. Document in README's "Getting started" section. Each fresh clone runs it once.
2. **Tooling like `pre-commit`** (Python) or `husky` (npm) — install hooks programmatically, but adds a dependency.
3. **CI as the safety net** (which is what caught it this time, so already in place).
